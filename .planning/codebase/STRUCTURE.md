## Directory Purposes

**apps/web:**

- Purpose: Next.js frontend application for user-facing itinerary generation UI
- Contains: React components, pages, state management, API client, hooks
- Key files: `src/app/page.tsx` (landing), `src/app/requirements/page.tsx` (intake form), `src/app/itinerary/[id]/page.tsx` (display)

**apps/functions:**

- Purpose: Netlify Functions backend API handling all server-side logic
- Contains: Authentication handlers, request processing, agent invocations, shared utilities
- Key files: `src/itineraries/process-request.ts` (orchestration), agent files, auth handlers

**packages/shared:**

- Purpose: Centralized location for types, config, and utilities used across frontend and backend
- Contains: TypeScript interfaces, Zod schemas, configuration management, constants
- Key files: `src/types/index.ts` (all type definitions), `src/config/index.ts` (env config)

**packages/database:**

- Purpose: Database client and schema definitions for Supabase persistence
- Contains: Supabase client setup, SQL migrations, type definitions
- Key files: `src/client.ts` (client setup), `src/types/database.ts` (schema types)

**packages/agents:**

- Purpose: Multi-agent system package for orchestrating itinerary generation
- Contains: Individual agent implementations and orchestration logic
- Key files: Directories for each agent type (research/, curation/, validation/, response/)

**docs:**

- Purpose: Project documentation including architecture decisions, requirements, stories, QA
- Contains: Architecture records, PRD, user story details, test plans

## Key File Locations

**Entry Points:**

- `apps/web/src/app/layout.tsx`: Root layout wrapper for all pages
- `apps/web/src/app/page.tsx`: Home page with navigation to forms
- `apps/web/src/app/requirements/page.tsx`: Multi-step intake form for user requirements
- `apps/functions/src/itineraries/process-request.ts`: Main API endpoint for processing requests

**Configuration:**

- `packages/shared/src/config/index.ts`: Centralized config with environment variable management
- `apps/web/next.config.js`: Next.js build configuration
- `apps/functions/build.js`: Esbuild bundler configuration for function compilation
- `netlify.toml`: Netlify deployment configuration
- `tsconfig.json`: TypeScript compiler options (shared across monorepo)

**Core Logic:**

- `apps/web/src/lib/api/itinerary.ts`: API client for all itinerary operations (submission, status, retrieval)
- `apps/web/src/stores/requirementsStore.ts`: Zustand store for form state management with persistence
- `packages/shared/src/types/index.ts`: Comprehensive type definitions for entire system
- `apps/functions/src/shared/response.ts`: Response helper utilities (success/error formatting)

**Testing:**

- `apps/web/src/__tests__/`: Frontend test files (co-located with source)
- `apps/functions/src/__tests__/`: Backend test files (integration and unit tests)
- `vitest.config.ts`: Test runner configuration

## Naming Conventions

**Files:**

- **Components:** PascalCase with .tsx extension (e.g., `RequirementsIntakeForm.tsx`, `ActivityCard.tsx`)
- **Utilities/Services:** camelCase with .ts extension (e.g., `itinerary.ts`, `auth.ts`, `logger.ts`)
- **Stores:** camelCase with Store suffix (e.g., `requirementsStore.ts`, `authStore.ts`)
- **Types/Schemas:** camelCase with .ts extension in `types/` and `schemas/` directories
- **Test files:** Matching source filename with .test.ts or .test.tsx suffix (e.g., `ActivityCard.test.tsx`)

**Directories:**

- **Plural for collections:** `components/`, `stores/`, `utils/`, `types/`, `schemas/`, `agents/`
- **Kebab-case for multi-word:** `form-steps/`, `auth-middleware/`
- **Feature-based organization:** `forms/`, `itinerary/`, `auth/` group related functionality

**TypeScript Types/Interfaces:**

- **Types:** PascalCase prefix with Type suffix (e.g., `PersonaType`, `ProcessingStatus`)
- **Interfaces:** PascalCase without suffix (e.g., `UserRequirements`, `ItineraryRequest`, `Activity`)
- **Enums:** Not used; discriminated unions with literal types instead (e.g., `ProcessingStatus`)

## Where to Add New Code

**New Feature Implementation:**

- **Frontend Page:** Create new folder in `apps/web/src/app/` (e.g., `new-feature/page.tsx`)
- **Frontend Component:** Create in `apps/web/src/components/` with subdirectory if grouped (e.g., `components/new-feature/MyComponent.tsx`)
- **Frontend Store:** Add to `apps/web/src/stores/` with naming pattern `{featureName}Store.ts`
- **Backend Handler:** Create new file in `apps/functions/src/{category}/` where category is auth/agents/itineraries
- **Shared Type:** Add interface to `packages/shared/src/types/index.ts`
- **Shared Schema:** Add Zod schema to `packages/shared/src/schemas/` in new or existing schema file

**New Component/Module:**

- Create in most specific subdirectory matching its purpose (e.g., form components in `components/forms/`)
- Always create corresponding test file in `__tests__/` mirror directory with same structure
- Export from index.ts barrel file if creating new subdirectory

**Utilities and Helpers:**

- **Frontend utilities:** `apps/web/src/lib/` for client-side helpers (API client, hooks, monitoring)
- **Backend utilities:** `apps/functions/src/shared/` for shared function utilities (auth, logging, response)
- **Cross-package utilities:** `packages/shared/src/utils/` for utilities used by multiple packages

## Special Directories

**apps/web/.next:**

- Purpose: Next.js build output and cache
- Generated: Yes (via `npm run build`)
- Committed: No

**apps/functions/dist:**

- Purpose: TypeScript compilation output
- Generated: Yes (via `npm run build`)
- Committed: No

**packages/\*/dist:**

- Purpose: Compiled TypeScript output for shared packages
- Generated: Yes (via `npm run build`)
- Committed: No

**node_modules:**

- Purpose: Installed dependencies for all packages
- Generated: Yes (via `npm install`)
- Committed: No

**.netlify/functions:**

- Purpose: Compiled Netlify Functions ready for deployment
- Generated: Yes (via build process)
- Committed: No

**docs/:**

- Purpose: Architecture decisions, requirements, stories, QA planning
- Generated: No (manually maintained)
- Committed: Yes

**.planning/codebase/:**

- Purpose: GSD-generated codebase analysis and planning documents
- Generated: Yes (via `/gsd:map-codebase` command)
- Committed: Yes (useful for future Claude instances)
